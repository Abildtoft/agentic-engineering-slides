# Raising the Bar for Juniors

<v-click>

**Before:** Juniors wrote code designed by others. Mid and senior engineers owned system design.

</v-click>
<v-click>

**Now:** Writing code is no longer the primary constraint. Syntax and frameworks are becoming interchangeable.

</v-click>
<v-click>

**The new focus:** Deciding what should exist, not just how to type it.

</v-click>

<!--
[T+122:00 | S5 slide 6 of 15 | 2min]

KEY POINTS:
- Old division of labour made sense when writing code was the bottleneck — it no longer is
- With AI, inference is fast and cheap — writing code in one stack today and another tomorrow is normal, code is increasingly disposable
- Both empowering and daunting: juniors can produce working software on day one, but the skills that matter (system design, problem decomposition, architectural judgment) traditionally took years of implementation experience
- This raises the bar — it doesn't lower it

# Clarity Merchants

<v-click>

**Shape ambiguous problems** into actionable intent

</v-click>
<v-click>

**Design the context architecture** that makes good outcomes inevitable

</v-click>
<v-click>

**Judge what matters** from what merely works

</v-click>

<!--
[T+126:45 | S5 slide 9 of 15 | 2min]

SOURCE: Addy Osmani, "The Best Engineers Never Just Wrote Code" (x.com/addyosmani/status/2007899127925854536)

KEY POINTS:
- Osmani's three capabilities of engineers who will thrive — walk through each:
  1. Problem formulation — take a messy situation and articulate what needs to be true; this is what S3's spec-driven development operationalises
  2. Context architecture — CLAUDE.md, AGENTS.md, documentation that shapes agent behaviour; not documentation for its own sake, but infrastructure for AI-assisted work
  3. Judgment — knowing the difference between "it works" and "it's right"; the skill cognitive debt erodes most
- For product/UX: every one applies — problem formulation is product work, context architecture is design systems, judgment is taste
- Osmani: "The craft evolves. It always has. But it remains craft."

BRIDGE: "Osmani calls it judgment. Paul Graham calls it something simpler."

# Most products don't die from missing features. They die from accumulation.

<!--
[T+128:45 | S5 slide 11 of 15 | 1min]

SOURCE: "On Subtraction" (x.com/ryolu_/status/2015824853121477198)

KEY POINTS:
- Adding is easy — someone asks for a feature, you build it; user hits a bug, you patch it; repeat until 50 buttons and no one knows where to start
- When building is cheap, the temptation to add is overwhelming — the cost of subtraction goes up, not down, because saying "no" to something nearly free to build requires conviction
- "Addition is momentum. Subtraction takes conviction. You have to see the whole, not just the parts."
- AI can build anything you ask for — it cannot tell you what to leave out
- For all roles: think about the last feature your team shipped that made the product worse by existing — that's what subtraction prevents

BRIDGE: "That subtraction instinct — let's make it concrete for product leaders."
